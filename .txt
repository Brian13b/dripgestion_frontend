## 🔴 Lo que falta completar

### 1. Modelo `Producto` (referenciado pero no compartido) - Si esta completo asi
El código usa `app.models.producto.Producto` en `logistica_service.py` pero no se compartió ese modelo.
- Verificar que tenga: `id`, `nombre`, `tenant_id`
- Si no existe, crearlo

### 2. Rutas de `clientes.py` — restricciones de rol
El router actual permite que cualquier usuario autenticado cree, edite o liste clientes.
Recomendación:
- `GET /clientes` → Admin y Repartidor (el repartidor necesita ver sus clientes)
- `POST /clientes` → Solo Admin - DEberia agregar a repartidor aca para que pueda agregar clientes por si se cruza algun cliente nuevo de camino
- `PUT /clientes/{id}` → Solo Admin - Al igual que aca tambien, ya que deberia el repartidor poder cambiar los datos de un cliente como el telefono cuando este le diga 
- `DELETE /clientes/{id}` → Solo Admin
- `GET /clientes/{id}` → Admin y Repartidor

### 3. Portal del cliente (rol CLIENTE) - Existen no te los pase nomas
Actualmente el rol existe pero no hay endpoints propios para el cliente.
El cliente debería poder:
- `GET /portal/mi-cuenta` → Su perfil (saldo, stock envases)
- `GET /portal/mis-movimientos` → Su historial de entregas

### 4. Endpoint de listado de repartidores (para el admin)
El admin necesita poder asignar repartidores a recorridos, pero no hay endpoint para listar usuarios con rol repartidor.
Agregar en el router de users:
- `GET /users?role=repartidor` → Lista de repartidores del tenant - Genial esto si hacia falta

### 5. Validación de `tenant_id` en `deps.py`
`get_current_user` no valida que el `tenant_id` del token coincida con el del usuario en DB.
Esto es un riesgo si el token fue generado con un tenant_id incorrecto.
Agregar la validación al decodificar el JWT. 

### 6. `dashboard_service.py` — cálculo de envases
El cálculo actual suma `sum(c.stock_envases.values())` que puede fallar si los valores
del dict no son numéricos (ej: si hay anidamiento). Agregar type guard.

### 7. Schema `__init__.py` 
Agregar `RecorridoAsignarRepartidor` al `__init__` de schemas para que sea importable
como `schemas.RecorridoAsignarRepartidor`.

## 💡 Nota para el frontend

El token JWT ya contiene el `role` del usuario. El frontend puede leerlo al hacer login
y usarlo para mostrar/ocultar secciones. **Pero el backend siempre valida el rol también**,
así que aunque el frontend no restrinja algo, el backend lo rechazará.

Flujo sugerido:
1. Al login, guardar `{ role, tenant_id, user_id }` del token decodificado
2. Usar el `role` para renderizar el menú y rutas disponibles
3. No deshabilitar botones según rol en el frontend es aceptable porque el backend
   retornará 403 de todas formas — pero mejora la UX hacerlo igual
